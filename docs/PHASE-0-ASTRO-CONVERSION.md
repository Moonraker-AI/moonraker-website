# Phase 0: moonraker.ai static HTML -> Astro (file-by-file plan)

Status: DRAFT for operator review (2026-06-07)
Parent: docs/CONTENT-SYSTEM.md
Goal: convert the 17-page hand-built static site to Astro WITHOUT changing what
the Cloudflare worker serves, so the deploy path, URLs, AI files, redirects, CSP,
and visual output are byte-for-byte equivalent. This is the foundation the blog
(Phase 1) renders into. No content pipeline, no blog, and no image-pipeline
change happen in Phase 0.

## 0. Non-goals (explicitly deferred)

- No `/blog/` routes yet (Phase 1).
- No Astro `<Image>` optimization: images stay verbatim in `public/assets/` with
  their existing hand-authored `<picture>` AVIF/PNG markup (changing this would
  alter paths + R2 keys and risk visual regressions). Future optimization.
- No DB / Supabase wiring.
- No worker changes (the worker is already correct; we only feed it the same R2
  layout from a different generator).

## 1. Hard constraints (the worker contract, worker/src/index.js)

The Astro build output, after publish, must keep ALL of these true:

1. R2 keys under prefix `moonraker-website/`: `index.html`,
   `<route>/index.html`, `<route>/index.md`, `assets/**`, `styles/**` (or the
   bundled CSS), `_astro/**`, `llms.txt`, `robots.txt`, `sitemap.xml`.
2. Clean URLs: every page is `<route>/index.html` (Astro `build.format:
   'directory'`). Root is `index.html`.
3. Markdown negotiation: every HTML page has a `.md` sibling at
   `<route>/index.md` (generated by build_r2.py from `<main>`).
4. Redirects stay in the worker (308: /contact, /privacy-policy,
   /terms-of-service; 301 trailing-slash strip; www -> apex). Astro emits none.
5. `/favicon.ico` -> `assets/logo.png` (worker handles; keep logo.png in
   public/assets).
6. CSP allow-list unchanged (worker): inline scripts/styles allowed
   ('unsafe-inline'), Google Fonts, Loom script, and the youtube/vimeo/spotify/
   apple/gamma/loom frame origins for Knowledge Hub embeds. Astro inline theme
   script + bundled styles are compatible.
7. `Link: </llms.txt>; rel="describedby", </sitemap.xml>; rel="sitemap"` on HTML
   (worker adds it; nothing to do in Astro).

## 3. Step-by-step

### 3.1 Scaffold (NEW files)
- `package.json`: `astro` dep; scripts: `dev` (astro dev), `build` (astro build),
  `preview` (astro preview), plus a `prepare` git-hooks line consistent with the
  other JS repos so `.githooks/pre-commit` (em-dash block) stays enabled.
- `astro.config.mjs`: `output: 'static'`, `build: { format: 'directory' }`,
  `trailingSlash: 'never'`, `site: 'https://moonraker.ai'`, NO integrations in
  Phase 0 (no React/island runtime; the site is pure HTML + vanilla site.js).
- `tsconfig.json`: extend `astro/tsconfigs/base`.
- Update `.gitignore`: add `dist/`, `.astro/`, `node_modules/` (confirm against
  the Dev `.gitignore-template`).

### 3.2 Move static assets into public/ (git mv, preserves history)
- `git mv assets public/assets`
- `git mv styles/globals.css src/styles/globals.css` (globals becomes an import)
- `git mv llms.txt robots.txt sitemap.xml public/`
- `assets/site.js` stays at `public/assets/site.js`, still referenced as
  `/assets/site.js`. NO edits to site.js (nav dropdowns, auto-hide, FAQ
  accordion, theme toggle, embed facade, reveal observer all keep working because
  class names are unchanged).

### 3.3 BaseLayout.astro (the single biggest win: kills nav/footer duplication)
Lift the EXACT `<head>` meta block, `<nav class="site-nav">`, and
`<footer class="site-footer">` markup verbatim from `index.html`. Structure:
- `<head>`: charset/viewport, per-page `<title>` + meta description + canonical +
  OG/Twitter from props, Google Fonts link, `import '../styles/globals.css'`, and
  the pre-paint theme `<script is:inline>` (reads localStorage `moonraker-theme`,
  sets `documentElement.classList` before first paint = no flash).
- `<body>`: nav, then `<main>{slot}</main>` (the `<main>` wrapper is REQUIRED:
  build_r2.py extracts `<main>` for the .md sibling), then footer, then
  `<script defer src="/assets/site.js">`.
- Props contract: `{ title, description, canonical, ogImage?, bodyClass? }`.
- Active nav state (today there is NONE): compute from `Astro.url.pathname` and
  add `aria-current="page"` + an `.is-active` class to the matching top-level
  link and its parent dropdown. Mapping in section 4.

### 3.4 Port each page (12 flat + 5 under core-marketing-system)
For each source `*/index.html`:
1. Create the matching `src/pages/...astro`.
2. Wrap in `<BaseLayout title=... description=... canonical=...>` using that
   page's existing `<title>`/meta/canonical/OG values (copy them exactly).
3. Move the page body (everything between `</nav>` and `<footer>`, inside a
   `<main>`) into the layout slot.
4. Move that page's inline `<head>` `<style>` block into a scoped `<style>` in
   the `.astro` page. Scoped styles keep the class names (Astro only adds a data
   attribute), so site.js selectors (`.partners-section`, `.core-os`, `.section`,
   `.cta-band`, `.nav-toggle`, FAQ classes) still match.
5. Funnel pages (book-a-call, free-strategy-call, reschedule) keep their
   scheduler embeds as-is; verify the embed origins are already in the worker CSP
   (they are for the current ones).

### 3.5 CSS strategy
- `globals.css`: imported once in BaseLayout -> Astro bundles + hashes it under
  `/_astro/`. Worker serves it immutable. (Drop the old `<link
  rel="stylesheet" href="/styles/globals.css">`.)
- Per-page inline styles: become scoped component `<style>` blocks (3.4 step 4).
- Recommended over keeping a verbatim `public/styles/globals.css` link because it
  gets minified + content-hashed caching, and Phase 0's parity gate is a visual
  diff anyway. If any scoped-style regression appears, fall back to a `:global()`
  wrapper (known Astro gotcha for child `<img>`, see memory).

### 3.6 AI files
Phase 0: keep `llms.txt`, `robots.txt`, `sitemap.xml` static in `public/`
(verbatim, no behavior change). Phase 1 replaces sitemap + llms with a build-time
generator (so blog posts auto-register), reusing the materialize.mjs helper
shapes from moonraker-site-template.

### 3.7 Delete vercel.json
Legacy and unused (worker is authoritative for headers/redirects). Remove it to
avoid future confusion.

### 3.8 Deploy path changes (scripts/)
- `publish_r2.sh`: prepend `npm ci && npx astro build` (produces `dist/`), THEN
  run the (retargeted) build_r2.py + upload_r2.py, THEN the worker deploy step
  (unchanged).
- `build_r2.py`: change `ROOT` to point at `dist/` and enumerate `dist/**`
  instead of the source tree. The `<main>` -> markdown logic is unchanged; it
  walks every `index.html` in `dist/` and writes `index.md` siblings into the
  staging dir, then the manifest maps `dist/...` + staged `.md` to
  `moonraker-website/...` R2 keys. upload_r2.py is unchanged (manifest-driven).
- Net effect: same R2 keys, same .md siblings, same worker. Only the generator
  changed from hand-HTML to Astro.

## 5. Acceptance / parity gate (must all pass before deploy)

Build locally (`npx astro build` + `npx astro preview`) and verify:

- [ ] All 17 routes return 200 at their clean URL.
- [ ] `dist/` contains `<route>/index.html` for every route (directory format).
- [ ] Visual diff vs current live moonraker.ai: each page matches (screenshot
      compare; pay attention to per-page sections now using scoped styles).
- [ ] Theme toggle: no flash on load, persists across pages.
- [ ] Nav: dropdowns open/close, auto-hide on scroll, mobile CTA injection, FAQ
      accordion, reveal-on-scroll all work (site.js unchanged).
- [ ] Active nav state highlights the right item per page (new behavior; sanity
      only, not a regression risk).
- [ ] Knowledge Hub embeds (youtube/spotify/etc.) still load under the worker CSP.
- [ ] All `/assets/**` (AVIF + PNG/webp fallbacks) resolve; logo + og-image load.
- [ ] Per-page `<title>`, meta description, canonical, OG tags match the originals.

After deploy to R2 (staging via workers.dev host, then apex), verify against the
worker:

- [ ] `curl -sI https://moonraker.ai/results` = 200; `/results/` 301 -> `/results`.
- [ ] 308 redirects: `/contact` -> `/book-a-call`, `/privacy-policy` -> `/privacy`,
      `/terms-of-service` -> `/terms`.
- [ ] `curl -H 'Accept: text/markdown' https://moonraker.ai/process` returns the
      `.md` sibling (markdown negotiation) with `Vary: Accept`.
- [ ] `/favicon.ico` returns the logo.
- [ ] `/llms.txt`, `/robots.txt`, `/sitemap.xml` present; HTML responses carry
      the `Link:` describedby/sitemap header.
- [ ] Run scripts/psi_audit.py (or the agent-ready-audit skill) to confirm no
      regression in PSI / agent-readiness Level 3.

## 7. Risks + mitigations

- Scoped-style regressions on per-page sections (esp. child `<img>` width/object
  -fit): mitigate with the `:global(img)` gate (known gotcha) and the visual diff.
- CSS load order: globals imported in BaseLayout loads before page-scoped styles,
  matching the current global-then-inline order. Verify cascade on the homepage
  (it has the most inline CSS).
- Astro inlining small scripts/styles: fine under the worker CSP ('unsafe-inline'
  is already allowed for both). No CSP change needed.
- build_r2.py retarget: confirm it walks `dist/` not source, or it will ship the
  OLD source HTML. Add a guard that errors if `dist/` is missing.
- `.md` extraction depends on `<main>`: every page MUST render inside BaseLayout's
  `<main>` or its .md sibling will fall back to the nav->footer slice.

## 8. Rollback

The current static site is the live source until the new R2 upload happens. If
the Astro build/parity fails, simply do not run the new publish_r2.sh; the
existing R2 objects keep serving. Git: do the whole conversion on a branch
(`astro-conversion`), merge only after the parity gate passes.

## 9. Order of operations

1. Branch `astro-conversion`.
2. Scaffold (3.1) + move assets (3.2). Commit.
3. BaseLayout (3.3) + port `index.astro` first; build + visual-diff the homepage
   (highest CSS complexity = best early signal). Commit.
4. Port remaining 16 pages (3.4). Build + diff. Commit.
5. CSS cleanup (3.5), delete vercel.json (3.7). Commit.
6. Deploy-script changes (3.8). Dry-run build_r2.py against dist/, inspect
   manifest + a sample .md. Commit.
7. Full parity gate (section 5) on local preview.
8. Publish to R2 (staging host first), worker-side parity gate.
9. Merge to main.
